# v1.7 Phase F/G/H Plan

This note organizes the next implementation sequence after placeholder AI index and read-only visibility.

## Goal

Advance Chronicle Stack from local placeholder adapter contracts toward an explicit local AI runtime vertical slice without collapsing key boundaries.

The boundaries remain:

- no hidden external runtime
- generated output is not primary record
- explicit/manual invocation before automation
- GUI mutation follows command-layer review, not the reverse

## Phase F: Local LLM provider adapter with explicit manual invocation

### Intent

Add a local provider adapter interface and an explicit command path for manual invocation.

### Candidate scope

- local provider contract model
- provider configuration surface under explicit runtime boundary
- manual command such as:
  - `chronicle runtime summarize --job ...`
  - or `chronicle summary run --id ...`
- output persisted as draft / pending review state
- explicit provenance fields for provider kind, model name, invocation mode, and external-call flag

### Non-goals

- auto-run on record creation
- hidden background workers
- implicit network access
- automatic approval

## Phase G: GraphRAG pipeline skeleton

### Intent

Introduce a skeleton pipeline shape without claiming a GraphRAG engine.

### Candidate scope

- query request model
- retrieval-plan model
- local placeholder retrieval composition across:
  - graph-ready export
  - placeholder vector results
  - selected Chronicle records
- dry-run command that shows what would be used
- optional package/export handoff contract for downstream runtime

### Current step now in repo

- `chronicle runtime retrieve-plan --record` persists a reviewable dry-run retrieval record
- local UI detail for `/api/runtime-records/<event_id>` now shows:
  - runtime preview summary
  - source hit counts
  - referenced record IDs
  - read-only downstream handoff contract with suggested CLI follow-ups
  - read-only package handoff preview for eligible context records
  - derived package review snapshot before persistence
- local UI detail for `/api/review-queue/<event_id>` now also shows package/export readiness as a read-only derived preview
- local UI list for `/api/review-queue` now surfaces readiness badges for faster triage

### Non-goals

- production retrieval quality claims
- ranking engine claims
- vector DB / graph DB runtime inclusion
- end-to-end autonomous reasoning loop

## Phase H: Mutation-capable GUI approval workflow with auth/authz design

### Intent

Only after command-layer review is mature, add a write-capable GUI path with explicit security framing.

### Candidate scope

- GUI preview of approve / reject / request-changes actions
- auth/authz design note before enabling write routes
- explicit mutation boundary and audit insertion points
- server-side write disable by default

### Non-goals

- open write API by default
- unaudited GUI mutation
- hosted multi-user service assumptions

## Recommended order

1. Phase F provider contract and explicit invocation
2. Phase F review-state persistence refinement
3. Phase G dry-run pipeline skeleton
4. Phase G downstream handoff / package boundary
5. Phase H auth/authz design note
6. Phase H gated GUI mutation preview

## Acceptance mindset

Each phase should preserve:

- primary record authority
- derived-surface rebuildability
- no hidden runtime expansion
- review before trust
- assistive output, not correctness proof
