# v1.7 Phase H Gated GUI Mutation Preview

Related: `docs/adr/0021-gated-local-ui-mutation-write-path.md`, `docs/v1.7-phase-h-auth-ui-design.md`, `docs/v1.7-phase-h-ui-mutation-threat-model.md`, `docs/v1.7-phase-h-readiness-status.md`

## Purpose

This note defines the concrete preview-stage contract for future GUI review mutation without enabling GUI mutation today.

It describes:

- the future route family
- required gating conditions
- request/response shape expectations
- fail-closed behavior
- audit insertion expectations

## Current status

Current repository state remains:

```text
read-only
preview-only
mutation disabled
CLI review authoritative
```

This document is therefore a design target, not an activation note.

## Proposed route family

If Phase H mutation is implemented, the local UI should use one route family only:

```text
POST /api/review-actions/<event_id>/approve
POST /api/review-actions/<event_id>/reject
POST /api/review-actions/<event_id>/request-changes
```

These routes should remain absent or fail closed until ADR-0021 preconditions are satisfied.

Current implementation now exposes this route family only as a fail-closed skeleton:

- supported actions: `approve`, `reject`, `request-changes`
- response status: `403 Forbidden`
- error code: `mutation_disabled`
- operator hint: `cli_equivalent` points back to the authoritative CLI review command

This keeps the repository on the Phase H read-only boundary while locking the HTTP contract for future enablement.

## Required request contract

Every future GUI mutation request should carry explicit local reviewer context:

```json
{
  "reviewer_label": "alice",
  "reviewer_kind": "local_operator",
  "session_label": "desk-session-1",
  "note": "optional review note",
  "ui_intent": "approve"
}
```

Minimum rules:

- `reviewer_label` is required
- `reviewer_kind` is required
- `session_label` is required when session-gated review is enabled
- `ui_intent` must match the route action
- request bodies must not be treated as sufficient proof of authority on their own

## Required preconditions

Before any route may mutate review state, all of these must hold:

1. `mutation_enabled == true` through explicit server-side configuration
2. authenticated local reviewer context is established
3. authorization check passes for the requested review action
4. target review row is still pending and eligible
5. CLI-equivalent action remains present in parity derivation
6. audit insertion path is available

If any precondition fails, the route must return a non-success result and persist no mutation.

## Expected success response

On success, the route should return descriptive confirmation only after mutation and audit insertion both complete:

```json
{
  "ok": true,
  "status": "applied",
  "event_id": "evt_...",
  "action": "approve",
  "audit_id": "aud_...",
  "decision_event_id": "evt_...",
  "cli_equivalent": "chronicle review approve --event evt_...",
  "mutation_enabled": true
}
```

## Expected failure response

When a mutation cannot be applied, the route should fail closed:

```json
{
  "ok": false,
  "status": "blocked",
  "event_id": "evt_...",
  "action": "approve",
  "error_code": "authorization_failed",
  "message": "Authorization requirement not satisfied.",
  "mutation_enabled": false
}
```

Typical `error_code` values:

- `mutation_disabled`
- `authentication_required`
- `authorization_failed`
- `review_not_pending`
- `cli_parity_drift`
- `audit_insertion_failed`
- `decision_persistence_failed`

## Fail-closed rule

Future GUI mutation must follow this transaction rule:

```text
no durable GUI review result is successful unless
review decision persistence and audit insertion both succeed
```

If either side effect fails:

- response is non-success
- durable mutation must not be reported as applied
- operator should be directed back to CLI recovery or inspection

## UI behavior expectations

Before enablement:

- buttons remain disabled
- auth-readiness notices stay descriptive
- parity/readiness summaries remain read-only

After enablement:

- the UI may reveal active buttons only for authorized actions
- route failures must surface exact blocker reasons
- CLI-equivalent commands should remain visible for auditability and recovery

## Non-goals

This preview note does not:

- enable GUI mutation now
- define remote or hosted identity systems
- replace CLI review as the baseline mutation contract
- define browser storage/session implementation details
- define final CSRF/session-cookie implementation

## Acceptance target

Phase H write-path implementation is ready to begin only when this preview contract, ADR-0021, and the threat-model controls all point to the same gated mutation shape.
