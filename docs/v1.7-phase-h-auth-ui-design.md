# v1.7 Phase H Auth and GUI Mutation Design Note

Related: `docs/adr/0016-http-bridge-auth-boundary.md`, `docs/adr/0018-local-ui-read-only-navigation-boundary.md`, `docs/adr/0019-local-ui-review-semantics-parity-boundary.md`, `docs/adr/0021-gated-local-ui-mutation-write-path.md`, `docs/adr/0022-explicit-local-ui-mutation-enable-flag.md`, `docs/adr/0023-browser-triggered-review-write-semantics.md`, `docs/v1.7-phase-h-ui-mutation-threat-model.md`, `docs/v1.7-phase-h-gated-gui-mutation-preview.md`, `docs/smoke-test-v1.7-phase-h.md`, `docs/v1.7-phase-h-readiness-status.md`

Repository-side closeout state for the broader v1.7 Phase F/G/H track is summarized in `docs/v1.7-phase-f-g-h-closeout-summary.md`.

## Intent

Phase H should not begin by adding write endpoints directly.

Instead, it should define the boundary for mutation-capable GUI review flows before any approve/reject/request-changes action is exposed from the local UI.

## Goals

- preserve command-layer review as the primary write path
- define authentication and authorization expectations before GUI mutation
- keep local-first semantics explicit
- add audit visibility for every future mutation path

## Required boundary

- GUI mutation must be disabled by default
- no hidden daemon or background worker
- no implicit multi-user server assumptions
- no write route without audit insertion points
- no write route without explicit auth/authz framing
- generated output remains assistive, not correctness proof

## Candidate design sequence

1. Document write-surface threat model
2. Define local auth/authz assumptions
3. Add explicit mutation capability flag to UI server config
4. Keep write routes unavailable by default
5. Add audit event requirements for approve/reject/request-changes
6. Add CLI-to-UI parity checks before enabling mutation preview

## Auth/authz questions

- Is the local UI single-user only by design?
- Is loopback binding sufficient, or is explicit session gating needed?
- How should local approval identity be represented in audit logs?
- What must happen when the UI is opened from a shared machine profile?

## Mutation preview before mutation enablement

Before enabling real writes, the UI can show:

- pending review queue
- action preview buttons
- exact CLI equivalent commands
- warning text describing that UI mutation is not enabled yet

This preserves operator visibility without weakening the current boundary.

Current implementation step:

- `/api/review-queue` exposes `needs_review` records as read-only preview rows
- detail view `/api/review-queue/<event_id>` remains preview-only by default and reveals active local-only mutation controls only behind explicit enablement
- `/api/ui-boundary` exposes mutation flag and auth/authz placeholder state
- UI host binding remains loopback-only until auth/authz exists
- UI mutation stays disabled
- `chronicle review` now provides append-only CLI parity for approve / reject / request-changes
- review CLI decisions now emit linked `review_decision` audit events
- review queue detail now exposes reviewer and audit timeline as read-only history
- review queue detail now exposes package/export readiness as a read-only derived preview
- review queue list now exposes package readiness summary badges before opening detail
- review queue list now also exposes CLI parity badges before opening detail, so preview-command drift remains visible in the list slice itself
- overview triage now aggregates those parity states too, so operators can jump straight into aligned or drift slices from the top-level read-only dashboard
- review queue sorting can now prioritize parity drift rows first, keeping contract drift visible without enabling mutation
- review queue list now also renders warning codes as badges, so common boundary blockers are visible before opening detail
- those warning badges are now clickable too, but only seed the same client-side read-only filter state
- overview triage now aggregates those warning codes too, so operators can jump straight into auth/authz blocker slices from the read-only dashboard
- those overview warning summaries are now ordered by boundary priority, keeping auth/authz blockers above weaker identity hints
- the same warning summaries now seed quick filters per warning code, but still only as client-side read-only navigation state
- when a warning-code filter is active, client-side sorting now also prioritizes rows carrying that warning before other attention heuristics
- the active-view summary now spells that out as `warning-first:<code>`, keeping the local sort rationale visible
- review queue list header now also shows the active slice as a removable chip, but still only as ephemeral local UI state
- runtime records list header now follows the same chip pattern, keeping read-only list state affordances consistent
- review/runtime active-slice chips now share one helper so the local UI vocabulary stays aligned across list surfaces
- active-view `filter=` / `sort=` labels now follow the same helper vocabulary, keeping list-state phrasing consistent
- overview triage button labels now follow the same helper vocabulary too, keeping counts and slices phrased consistently
- overview warning-priority badges now follow that same helper vocabulary as well
- review queue row badges for capability/readiness/parity now also come from dedicated helpers, reducing label drift across read-only surfaces

- related-link labels inside detail payloads now also share helpers, keeping `Open matching ...` and `Open context ...` phrasing aligned with the read-only UI vocabulary
- overview panel titles, detail notice titles, and triage summary JSON lines now also share helpers, reducing remaining heading/summary drift across the read-only UI
- detail notice field rows now also share helpers for plain values, joined lists, and JSON summaries, keeping payload-derived wording aligned across read-only surfaces
- ADR-0019 now captures the semantic rule behind those helpers: review wording, parity summaries, and mutation-capability signaling stay descriptive and non-activating until a later write-path ADR exists
- overview now exposes read-only triage aggregates so operators can prioritize review before opening queue rows
- overview triage now offers quick-action navigation into read-only runtime/review/package surfaces without enabling mutation
- runtime/review list filtering remains client-side and read-only; it changes visibility only, not records or server state
- overview quick actions may seed those client-side filters, but still only affect local view state
- related-link navigation between read-only detail views remains local UI state only and does not enable mutation
- breadcrumb/back navigation also remains client-side only and does not alter records or approval state
- breadcrumb context may include active filter text, but that remains local UI state only

- short trail previews of that history are likewise local-only UI context, not durable record metadata

- human-friendly trail labels improve readability only; they do not change record semantics or state
- related-link labels follow the same readability rule and remain derived UI presentation only

- client-side sort toggles follow the same rule and only reorder local presentation
- related-list shortcuts follow the same rule and only reopen derived read-only list views
- active-view summaries likewise reflect only ephemeral local UI state and derived navigation context
- inline list badges follow the same rule and summarize already-derived fields without mutating or certifying records
- badge-triggered filter seeds and no-match empty states also remain local UI behavior only
- review detail now exposes disabled approve/reject/request-changes preview controls with exact CLI equivalents, while keeping GUI mutation off

- the same action preview/result surfaces now also expose rollback status, possible error codes, and CLI recovery hints so fail-closed semantics stay visible at the point of action
- summary jobs list now also exposes matching reviewer identity / session assurance as a read-only derived badge surface
- local UI mutation is now guarded by a second explicit operator flag (`--enable-ui-mutation`) in addition to `--mutation-capability-flag`
- reviewer identity is now structured as label/kind/session for future auth/authz linkage
- review detail now derives identity assurance against the current UI boundary
- review detail warning/capability surfaces now also offer related read-only queue slices for the same blocker codes
- review detail parity / assurance / package-readiness notices now follow the same pattern with related read-only queue slices
- action preview notice now follows the same pattern with capability/parity-related read-only queue slices
- review timeline rows now follow the same pattern with disposition/identity-assurance-related read-only queue slices
- `chronicle ui --auth-mode/--authorization-mode` now provides placeholder boundary config for assurance transitions
- `/api/ui-boundary` and overview now also expose a derived `auth_boundary_summary`, so auth/authz placeholder blockers and next steps are visible without opening review detail first
- overview now also exposes a derived `auth_boundary_overview`, so auth-warning / authorization-warning / missing-identity slices are visible before opening review detail rows
- overview now also exposes a derived `identity_boundary_summary`, so reviewer identity / session alignment can be inspected without opening individual review detail rows

- explicit local mutation enablement semantics are now captured in ADR-0022 so preview intent and write enablement remain distinct

## Non-goals

- hosted review service
- background sync
- remote auth provider integration
- unattended approval automation
- changing primary-record authority away from Chronicle JSONL
