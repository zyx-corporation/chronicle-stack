# v1.7 Phase H Auth and GUI Mutation Design Note

## Intent

Phase H should not begin by adding write endpoints directly.

Instead, it should define the boundary for mutation-capable GUI review flows before any approve/reject/request-changes action is exposed from the local UI.

## Goals

- preserve command-layer review as the primary write path
- define authentication and authorization expectations before GUI mutation
- keep local-first semantics explicit
- add audit visibility for every future mutation path

## Required boundary

- GUI mutation must be disabled by default
- no hidden daemon or background worker
- no implicit multi-user server assumptions
- no write route without audit insertion points
- no write route without explicit auth/authz framing
- generated output remains assistive, not correctness proof

## Candidate design sequence

1. Document write-surface threat model
2. Define local auth/authz assumptions
3. Add explicit mutation capability flag to UI server config
4. Keep write routes unavailable by default
5. Add audit event requirements for approve/reject/request-changes
6. Add CLI-to-UI parity checks before enabling mutation preview

## Auth/authz questions

- Is the local UI single-user only by design?
- Is loopback binding sufficient, or is explicit session gating needed?
- How should local approval identity be represented in audit logs?
- What must happen when the UI is opened from a shared machine profile?

## Mutation preview before mutation enablement

Before enabling real writes, the UI can show:

- pending review queue
- action preview buttons
- exact CLI equivalent commands
- warning text describing that UI mutation is not enabled yet

This preserves operator visibility without weakening the current boundary.

Current implementation step:

- `/api/review-queue` exposes `needs_review` records as read-only preview rows
- detail view `/api/review-queue/<event_id>` remains read-only
- `/api/ui-boundary` exposes mutation flag and auth/authz placeholder state
- UI host binding remains loopback-only until auth/authz exists
- UI mutation stays disabled
- `chronicle review` now provides append-only CLI parity for approve / reject / request-changes
- review CLI decisions now emit linked `review_decision` audit events
- review queue detail now exposes reviewer and audit timeline as read-only history
- review queue detail now exposes package/export readiness as a read-only derived preview
- review queue list now exposes package readiness summary badges before opening detail
- overview now exposes read-only triage aggregates so operators can prioritize review before opening queue rows
- overview triage now offers quick-action navigation into read-only runtime/review/package surfaces without enabling mutation
- runtime/review list filtering remains client-side and read-only; it changes visibility only, not records or server state
- reviewer identity is now structured as label/kind/session for future auth/authz linkage
- review detail now derives identity assurance against the current UI boundary
- `chronicle ui --auth-mode/--authorization-mode` now provides placeholder boundary config for assurance transitions
- review queue now derives capability/warning surface from boundary + reviewer identity conditions
- CLI guidance is surfaced as `suggested_cli_family` until full approve/reject CLI parity exists

## Non-goals

- hosted review service
- background sync
- remote auth provider integration
- unattended approval automation
- changing primary-record authority away from Chronicle JSONL
