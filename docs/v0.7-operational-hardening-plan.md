# v0.7 Operational Hardening Plan

Status: active implementation plan  
Parent: #156  
Theme: Operational Hardening and Verified Context Workflows

## Purpose

v0.7 turns the advisory warnings observed in v0.6 release smoke into local, inspectable workflows.

The release does not introduce a server, daemon, model runtime, GraphRAG engine, vector DB, graph DB, or external service dependency. It improves the local CLI path for preparing context-use and derived-output workflows.

## Implemented workflow surfaces

### Context classification metadata

Primary CLI path:

```bash
chronicle context classification missing
chronicle context classification set --context <id> --layer internal --sensitivity internal
chronicle context classification show --context <id>
```

The classification workflow records a new Context snapshot with the same `context_id`. The derived index resolves to the latest classified snapshot. Classification metadata remains advisory metadata.

### Audit events

Primary CLI path:

```bash
chronicle audit record --operation export --purpose "review" --target local
chronicle audit list
chronicle audit show --id <audit-id>
```

Audit events are stored in `.chronicle/audit.jsonl`. They improve local traceability for export, context-use, reinterpretation, and package workflows.

### Lifecycle markers

Primary CLI path:

```bash
chronicle lifecycle record --target <id> --target-kind context --action seal
chronicle lifecycle list
chronicle lifecycle show --id <lifecycle-id>
```

Lifecycle events are stored in `.chronicle/lifecycle.jsonl`. They are advisory markers for downstream workflows and do not alter primary Chronicle records by themselves.

### Doctor guidance

`chronicle doctor` recommendations now point to the classification, audit, and lifecycle workflows where appropriate.

### Release smoke profile

The v0.7 smoke profile is documented in:

It demonstrates classification, audit, lifecycle, doctor, and derived export paths.

## Non-goals

v0.7 does not:

- implement a server or daemon
- implement model API calls
- implement GraphRAG execution
- require vector DB or graph DB dependencies
- claim formal correctness proof
- turn advisory metadata into enforcement
