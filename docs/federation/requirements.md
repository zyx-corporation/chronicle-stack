# Chronicle Stack 分散連合拡張 要求仕様

Status: Draft  
Scope: Chronicle Stack federation extension  
Author: Tomoyuki Kano

## 1. 目的

Chronicle Stack 分散連合拡張の目的は、Chronicle Stack を単なる分散投稿基盤ではなく、AI時代における文脈主権型SNSとして成立させることである。

従来のSNSは、主に発話、投稿、反応の流通を扱ってきた。しかしAI時代には、発話や文書の多くがAIによって要約、補筆、翻訳、分類、再構成される。その結果、完成物だけを保存しても、その発話が誰の問いから始まり、どの文脈を背景に持ち、どのAI処理を経由し、どの判断によって採用、修正、棄却されたのかが失われる。

Chronicle Stack は、この喪失に対抗するための基盤である。中心要求は次の一文に要約される。

> 各主体が文脈正本をローカルに保持し、共有可能な意味だけを、来歴・権限・撤回可能性・監査証跡付きで連合できること。

Chronicle Stack が扱う対象は、投稿そのものではなく、投稿や判断や生成物に至る来歴である。すなわち Chronicle Stack は、発話の流通基盤ではなく、意味形成の年代記を保存、検証、継承するための分散連合基盤である。

## 2. 基本概念

Chronicle Stack における基本単位は、アカウントではなく Node である。

Node は、個人、家庭、企業、士業、地域、研究組織、公共機関、プロジェクトなど、文脈主権を持つ主体を表す。各 Node は、自身の文脈正本を保持し、必要に応じて他 Node と Chronicle を共有、参照、撤回、監査、再構成する。

Chronicle Stack の中心データは投稿ではなく Chronicle オブジェクトである。Chronicle オブジェクトは、問い、会話、判断、生成物、差分、反論、仮説、撤回、減衰などを来歴付きで表現する単位である。

Chronicle Stack は、次の三つを区別する。

- 正本: 個人、企業、地域、団体などのローカルノードに保持される原文脈。
- 共有文脈: 他ノードに渡してよいと判断された、抽象化、匿名化、要約、証跡化、または範囲制御された文脈。
- 派生文脈: AI、人間、他ノードの処理によって生成された再構成物。必ず元文脈との差分および来歴を持つ。

## 3. 設計原則

### 3.1 ローカル正本原則

文脈の正本は、個人、企業、地域、団体などのローカルノードに置く。中央サーバーは文脈の正本を保持しない。中央的機能が存在する場合でも、その役割はインデックス、配送、発見、合意済み共有の補助に限定される。

### 3.2 最小共有原則

共有されるべきものは、生データではなく、目的に必要な最小文脈である。文書全文、会話全文、顧客情報、個人情報、未検証の心理推測などを無制限に共有してはならない。

共有対象は、必要に応じて、要約、証跡、判断理由、抽象化された知見、匿名化された失敗パターン、公開可能な根拠などに変換されるべきである。

### 3.3 来歴必須原則

すべての Chronicle オブジェクトは、誰が、いつ、どの問いから、どの根拠で、どの人間またはAI処理を経て、現在の形になったかを記録しなければならない。

Chronicle Stack において、来歴を持たない生成物は信頼対象ではなく、未検証の断片として扱われる。

### 3.4 可逆・撤回・減衰原則

共有は永久固定ではない。共有済み Chronicle は、条件に応じて撤回、期限切れ、範囲縮小、再分類、減衰、非公開化できなければならない。

これは単なる「忘れられる権利」ではなく、文脈が古くなり、誤解され、過剰一般化され、収奪されることを防ぐための権利である。

### 3.5 AI非依存原則

Chronicle Stack は、特定AIモデル、特定AI事業者、特定推論方式に依存してはならない。

AIは要約器、分類器、推論エンジン、検索補助、監査補助として利用できる。しかしAIは文脈の所有者ではない。文脈の所有者は Node であり、AIは交換可能な処理系として扱われる。

### 3.6 RDE監査原則

AIまたは人間による変換が行われた場合、元文脈から何が保存され、何が変換され、何が補完され、何が逸脱したかを記録できなければならない。

RDE、Resonant Deviation Evaluator、すなわち共鳴的逸脱評価は、Chronicle Stack における意味変化監査の中核である。

### 3.7 分散連合原則

Chronicle Stack は単一中央SNSではない。個人ノード、家庭ノード、企業ノード、士業ノード、地域ノード、研究ノード、公共ノード、プロジェクトノードが、権限と合意に基づいて相互接続する分散連合基盤である。

連合は、単なる接続ではなく、文脈境界、信頼関係、共有目的、撤回可能性を伴う関係でなければならない。

## 4. 主体モデル

Chronicle Stack の主体は Node として表現される。基本種別は次のとおりである。

- 個人ノード: 個人の会話、判断、文書、生成物、思考来歴を保持する。
- 家庭ノード: 家庭内の生活文脈、介護、教育、家族内合意、家計、生活判断などを保持する。
- 企業ノード: 業務判断、商談、契約、顧客文脈、経理、補助金、業務改善、AI番頭連携などを保持する。
- 士業ノード: 診断士、税理士、弁護士、行政書士、社労士などの専門的助言、監査、文脈補助を保持する。
- 地域ノード: 地域課題、住民合意、行政連携、災害対応、地域知識を保持する。
- 研究ノード: 研究仮説、論文、実験記録、反論、検証過程を保持する。
- 公共ノード: 公開政策、行政判断、公共記録、説明責任を保持する。
- プロジェクトノード: 特定プロジェクトの問い、議論、設計、判断、成果物、差分を保持する。

各 Node は、最低限、ローカル保存、権限管理、Chronicle オブジェクト作成、外部AI利用時の文脈境界管理、共有要求の承認・拒否・撤回、受信 Chronicle の検証、RDE監査結果の保持、他ノードとの連合接続を実行できなければならない。

重要なのは、Node が単なるサーバーではなく、文脈主権の単位であるという点である。

## 5. Chronicle オブジェクト仕様

Chronicle オブジェクトは、Chronicle Stack における中心データである。これは、出来事、問い、会話、判断、生成物、差分、反論、仮説、撤回、減衰などを来歴付きで表現する。

### 5.1 基本型

- Question Chronicle: 問いの発生、背景、未解決性を記録する。
- Conversation Chronicle: 人間同士、または人間とAIの対話を記録する。
- Decision Chronicle: なぜその判断に至ったかを記録する。
- Artifact Chronicle: 文書、コード、画像、設計書、提案書などの生成物と来歴を記録する。
- Delta Chronicle: 生成物や判断が元文脈からどう変化したかを記録する。
- Objection Chronicle: 反論、異議、留保、再審要求を記録する。
- Hypothesis Chronicle: 未検証仮説を事実と区別して保存する。
- Decay Chronicle: 期限切れ、忘却、撤回、非公開化、文脈劣化を管理する。

### 5.2 必須メタデータ

各 Chronicle オブジェクトは、最低限、次のメタデータを持たなければならない。

- ID
- 作成者または作成ノード
- 作成日時、更新日時
- 由来となる問い
- 根拠資料
- 関連する人物、組織、プロジェクト
- 公開範囲
- 機密区分
- AI関与の有無
- AIモデルまたは処理系の識別子
- 人間による採用、修正、棄却の記録
- RDE監査結果
- 撤回、減衰、保存期間ポリシー
- 署名または改ざん検知情報

### 5.3 AI解釈の扱い

Chronicle Stack は、AIの解釈を事実として保存してはならない。

AIによる心理推測、性格推定、思想推定、能力推定、意図推定、リスク推定などは、恒久的なユーザー属性として扱ってはならない。保存する場合でも、必ず「AI解釈」「仮説」「未検証」として分類し、減衰、再審、削除、撤回の対象にしなければならない。

ユーザー発言、事実記録、AI応答、AI解釈、人間判断、合意済み決定、未解決論点は、データ分類上明確に区別されなければならない。

## 6. 権限・同意・共有仕様

Chronicle Stack の分散連合は、権限モデルによって成立する。

通常のSNSにおける「公開・非公開」だけでは不十分である。Chronicle Stack では、公開範囲、共有目的、再共有可否、保存期間、撤回条件、フィールド単位権限を扱う必要がある。

### 6.1 共有範囲

- Private: 自ノード内のみで利用される。
- Trusted: 明示的に信頼した相手のみが利用できる。
- Project: 特定プロジェクト参加者のみが利用できる。
- Organization: 組織内のみで利用される。
- Community: 地域または団体内のみで利用される。
- Federated: 指定された連合先に限定して共有される。
- Public: 完全公開される。
- Expired: 期限切れにより非公開化された状態である。
- Revoked: 撤回済みの状態である。
- Tombstone: 本文を消去し、存在痕跡のみ保持する状態である。

### 6.2 フィールド単位権限

Chronicle Stack は、オブジェクト単位だけでなく、フィールド単位で共有範囲を制御できるべきである。

たとえば、判断理由は共有するが顧客名は共有しない。失敗パターンは共有するが契約書原本は共有しない。RDE監査結果は共有するが会話全文は共有しない。AI関与ラベルは公開するが、プロンプト全文は非公開にする。

### 6.3 共有要求

共有には必ず目的が必要である。目的のない共有は、文脈収奪へ変質する危険がある。

共有要求には、共有目的、共有範囲、対象フィールド、再共有可否、保存期間、匿名化・仮名化レベル、第三者提供制限、撤回条件が含まれなければならない。

### 6.4 撤回と派生物

撤回要求は、共有先ノードに送信できなければならない。

撤回後の扱いは、完全削除、非公開化、Tombstone 化、参照停止、派生物への警告付与、再審要求などから選択可能であるべきである。

派生 Chronicle が存在する場合、撤回は派生物へ伝播しなければならない。ただし、すでに独立した公共的記録となっている場合など、撤回不能または限定撤回となる条件も明示されなければならない。

## 7. 分散連合プロトコル要求

Chronicle Stack の分散連合には、通信プロトコルだけでなく、意味プロトコルが必要である。

通常の分散SNSであれば、投稿の配送が中心となる。しかし Chronicle Stack では、届ける対象が問いなのか、判断なのか、反論なのか、監査結果なのか、撤回要求なのか、共有同意なのかを識別できなければならない。

### 7.1 連合メッセージ

- Publish Chronicle: Chronicle を公開または共有する。
- Request Context: 文脈共有を要求する。
- Grant Context: 文脈共有を許可する。
- Deny Context: 文脈共有を拒否する。
- Revoke Context: 共有済み文脈の撤回を要求する。
- Update Chronicle: Chronicle の更新を通知する。
- Fork Chronicle: 派生 Chronicle の作成を通知する。
- Reference Chronicle: 他 Chronicle への参照を通知する。
- Object Chronicle: 異議申し立てを行う。
- Audit Chronicle: RDE監査結果を共有する。
- Decay Notice: 期限切れ、減衰、非公開化を通知する。
- Trust Assertion: 信頼表明を行う。
- Trust Withdrawal: 信頼撤回を行う。

### 7.2 意味関係

Chronicle Stack において Chronicle は単に流通するのではなく、関係を持つ。

引用、派生、反論、補完、再審、監査、撤回、減衰、合意、棄却、保留といった意味関係は、第一級オブジェクトとして扱われるべきである。

## 8. 信頼モデル

Chronicle Stack は、全主体を単一中央IDで管理しない。分散ID、署名、鍵管理、信頼グラフを用いる。

### 8.1 識別と署名

Chronicle Stack は、ノードID、主体ID、鍵管理、署名、公開鍵ローテーション、失効証明を扱える必要がある。

v0.1では、署名付きJSON、Git履歴、ローカルDB、エクスポート可能な証跡で十分である。DID、Decentralized Identifier、分散識別子、PoP、Proof of Personhood、ZKP、Zero Knowledge Proof、ゼロ知識証明は、将来拡張可能な抽象化として残す。

### 8.2 文脈別信頼

信頼は単一スコアではない。

ある相手を技術判断では信頼するが、政治判断では信頼しない場合がある。ある企業を契約実務では信頼するが、地域課題では限定的に扱う場合もある。

したがって Chronicle Stack の信頼モデルは、文脈別信頼でなければならない。信頼レベル、関係種別、権限委譲、代理投稿、代理判断、AI代理生成の明示が必要である。

## 9. AI連携要求

Chronicle Stack はAI時代の基盤であるが、AIに従属してはならない。

AI連携においては、外部AIへ送る文脈を最小化し、機密情報を検出し、匿名化・仮名化し、送信前にプレビューできなければならない。

また、モデル選択、モデル変更可能性、プロンプトと応答の保存可否設定、AI関与ラベル、AI要約と原文の分離、AI推測とユーザー発言の分離、RDE監査、未検証主張の検出、過剰断定の検出、外部AI停止時のローカルフォールバックが必要である。

Sayane は、ユーザー、外部AI、Chronicle Stack の間に立つ文脈制御層として位置づけられる。Sayane は、外部AIに何を渡すか、何を渡さないか、戻ってきた出力が元文脈をどう変形したかを監査する。

AI番頭では、この構造が業務向けに展開される。社長の判断、顧客文脈、商談、契約、補助金、経理、業務改善などを扱うため、Chronicle Stack との接続は、将来的には Enterprise 段階の拡張として設計されるべきである。

ただし MVP では、小規模実用に絞る。

## 10. データ保存要求

v0.1では、過剰な分散基盤よりも、ローカルファーストを優先する。

推奨保存構成は、Markdown、JSONL、SQLite、PostgreSQL、Git、Vector DB、Object Storage、署名付きManifestである。

- Markdown: 人間可読な本文を扱う。
- JSONL: イベントログを扱う。
- SQLite: 個人ノードまたは小規模ノードの管理DBとして使う。
- PostgreSQL: 組織ノード、地域ノード、研究ノードなどの共有基盤として使う。
- Git: 変更履歴、差分、レビュー可能性を担う。
- Vector DB: 検索補助として使う。
- Object Storage: 添付ファイルを保持する。
- 署名付きManifest: 改ざん検知を担う。

重要なのは、どれか一つに閉じないことである。人間可読性、機械処理性、履歴性、検索性、移植性を分けて考える。

保存対象は、事実、ユーザー発言、AI応答、AI解釈、合意済み決定、未解決論点、保留仮説、破棄対象、減衰対象、撤回対象に分類されなければならない。

何でも保存するシステムは、やがて監視装置になる。Chronicle Stack は保存基盤であると同時に、保存しないこと、減衰させること、撤回することを設計に含めなければならない。

## 11. 検索・再構成要求

Chronicle Stack の価値は保存ではなく再構成にある。

必要な検索機能は、全文検索、意味検索、時間軸検索、問い単位検索、人物・組織・プロジェクト検索、判断履歴検索、差分検索、反論検索、未解決論点検索、根拠資料検索、AI関与検索、RDE逸脱検索である。

検索結果は、単なる答えとして提示されるべきではない。来歴として再構成されるべきである。

たとえば「なぜこの方針になったのか」と問われた場合、Chronicle Stack は該当文書だけを返すのではなく、最初の問い、途中の議論、採用された案、棄却された案、AIが生成した案、人間が修正した箇所、未解決論点、反論、再審余地を時系列で提示する。

これは通常のRAG、Retrieval-Augmented Generation、検索拡張生成とは異なる。RAGは答えを作るために文書を探す。Chronicle Stack は、判断の来歴を再構成するために文脈を探す。

## 12. 文脈SNS機能要求

Chronicle Stack がAI時代のSNSになるためには、ソーシャル機能が必要である。ただし、既存SNSの模倣であってはならない。

Chronicle Stack は、注意経済ではなく、理解経済のUIを持つべきである。

必要な機能は、人をフォローする、問いをフォローする、プロジェクトをフォローする、地域課題をフォローする、Chronicle を購読する、反論を付ける、補足を付ける、再審要求を出す、派生 Chronicle を作る、共同 Chronicle を編集する、公開範囲ごとに発信する、AI関与ラベル付きで投稿する、RDE監査付きで公開する、である。

「いいね」や「リポスト」は慎重に扱うべきである。代替的な反応として、理解した、保留する、根拠不足、再審したい、別文脈では異なる、自ノードに取り込む、参照する、反論する、共同編集を提案する、といった意味ある反応を中心に設計する。

Chronicle Stack におけるSNS性は、承認欲求や拡散速度ではなく、問い、判断、反論、来歴、再審可能性によって成立する。

## 13. UI要求

Chronicle Stack のUIは、タイムライン中心ではなく、来歴中心であるべきである。

必要なビューは、Timeline View、Lineage View、Decision View、Delta View、Evidence View、Objection View、Trust View、Context Boundary View、AI Involvement View、Decay View である。

- Timeline View: 時系列を表示する。
- Lineage View: 何から何が生まれたかを表示する。
- Decision View: 判断履歴を表示する。
- Delta View: 意味変化を表示する。
- Evidence View: 根拠資料を表示する。
- Objection View: 反論、異議、留保、再審要求を表示する。
- Trust View: ノード間の信頼関係を表示する。
- Context Boundary View: 外部共有境界を表示する。
- AI Involvement View: AI関与履歴を表示する。
- Decay View: 期限切れ、撤回、忘却、減衰対象を表示する。

特に Lineage View と Delta View は重要である。これらがなければ、Chronicle Stack は高機能ノート、文書管理、または分散SNSに留まる。Chronicle Stack らしさは、「どこから来て、何が変わったか」を可視化する点にある。

## 14. セキュリティ要求

Chronicle Stack は文脈主権を扱うため、通常のSNSより機密度が高い。

最低限必要なセキュリティ要求は、保存時暗号化、通信時暗号化、ノード間署名検証、アクセス制御、フィールド単位権限、監査ログ、鍵ローテーション、端末紛失時の失効、バックアップ暗号化、外部AI送信前の機密検査、PII、Personally Identifiable Information、個人識別情報の検出、秘密情報の自動ラベル、安全なエクスポート、安全な削除・撤回、管理者権限の分離である。

特に、外部AIへの送信境界は必須である。Chronicle Stack の中に貴重な文脈が蓄積されるほど、それを外部AIにまとめて送ってしまう誘惑が強くなる。ここを Sayane が制御する必要がある。

## 15. 非機能要求

Chronicle Stack は、ローカルファーストであり、オフラインでも最低限使え、エクスポート可能で、ベンダーロックインせず、人間可読な形式を持ち、小規模ノードで動き、大規模連合に拡張でき、モデル非依存であり、権限モデルが後から拡張でき、監査ログが改ざん検知可能であり、日常利用で重すぎず、共有操作が直感的でなければならない。

v0.1では、分散連合の完全実装よりも、ローカル正本、来歴、差分、エクスポート、最小AI連携を優先する。

## 16. MVP要求

Chronicle Stack v0.1 のMVPは、「会話・文書・判断・生成物・差分を、後から再構成できる最小中核」とする。

v0.1必須要求は、ローカルChronicle保存、Chronicleオブジェクトの基本型、Markdown + JSONL保存、Gitまたはイベントログによる差分、問い・判断・生成物の関連付け、AI関与ラベル、簡易RDEメモ、検索、エクスポート、手動共有パッケージ作成、権限ラベル、減衰・撤回メモである。

v0.1では、GraphRAG、Dashboard、完全分散連合、DID、PoP、ZKPは後続とする。これらを最初から入れると、思想は大きくても実装が進まなくなる。

v0.2以降で、連合プロトコル、署名、ノード間共有、RDE自動支援、AI番頭連携、Sayane連携を拡張する。

v0.2.0以降のライセンス方針として、AGPL-3.0-or-later + 商用ライセンスのデュアルライセンスを採用する場合、本要求仕様はその境界設計とも整合する。ローカル正本・文脈主権・連合参加者の自由を担保しつつ、企業利用には商用ライセンスを提供できる。

## 17. レイヤー構成

Chronicle Stack 分散連合拡張は、次のレイヤーで整理される。

- Core Layer: 記録中核。問い、会話、判断、生成物、差分を保存する。
- Lineage Layer: 来歴層。何から何が生まれたかを追跡する。
- Context Boundary Layer: 文脈境界層。何を外部AIや他ノードに渡すかを制御する。
- RDE Layer: 意味監査層。元文脈からの保存、変換、補完、逸脱を監査する。
- Federation Layer: 分散連合層。ノード間で Chronicle を共有、参照、撤回、反論する。
- Trust Layer: 信頼層。署名、権限、信頼関係、主体性を管理する。
- Social Layer: 文脈SNS層。問い、論点、プロジェクト、地域課題を中心に人と文脈をつなぐ。
- AI Adapter Layer: AI接続層。外部AI・ローカルAIを交換可能な推論資源として接続する。

## 18. 開発ロードマップ案

v0.1 は、ローカルChronicle中核を実装する。対象はローカル保存、Chronicle基本型、Markdown + JSONL、Gitまたはイベントログ、問い・判断・生成物・差分の関連付け、AI関与ラベル、簡易RDEメモ、検索、エクスポート、手動共有パッケージである。

v0.2 は、Sayane連携とRDE強化を実装する。対象は外部AI送信境界、機密検査、AI要約と原文の分離、RDE差異メモの構造化、署名付きManifest、共有パッケージ検証である。

v0.3 は、分散連合プロトコルの初期実装を行う。対象は Publish Chronicle、Request Context、Grant Context、Deny Context、Revoke Context、Update Chronicle、Fork Chronicle、Reference Chronicle、Audit Chronicle の最小実装である。

v0.4 は、文脈SNS機能を導入する。対象は問いフォロー、Chronicle購読、反論、補足、再審要求、共同Chronicle編集、RDE監査付き公開である。

v0.5 は、信頼モデルと文脈別権限を強化する。対象はノードID、主体ID、鍵ローテーション、文脈別信頼、代理投稿、AI代理生成明示、フィールド単位権限である。

v1.0 は、分散連合型の文脈主権SNSとしての安定版を目指す。対象は複数ノード間連合、撤回伝播、減衰管理、UI統合、AI Adapter の安定化、監査可能なエクスポートである。

## 19. RDE差異検証

保存された要素は、Chronicle Stack が単なる知識管理ではなく、判断と意味形成の年代記を継承する基盤であるという思想である。特に、発話ではなく発話に至る来歴を扱うという中心命題は維持されている。

変換された要素は、従来のローカル文脈保存構想を、分散連合型SNSの基本要求仕様へ展開した点である。これにより、Chronicle Stack は個人または組織内の記録基盤から、個人、企業、士業、地域、研究ノードが相互接続する社会的記憶基盤へと拡張された。

補完された要素は、Node モデル、Chronicle オブジェクト型、権限・同意・共有仕様、連合メッセージ、信頼モデル、AI連携、UI要求、MVP境界、開発ロードマップである。

未解決の要素は、ActivityPub 系に寄せるのか、独自プロトコルを定義するのか、あるいは当初はエクスポート同期から始めるのかという実装選択である。また、DID、PoP、ZKPをどの段階で導入するかも未確定である。

逸脱リスクは、分散連合を急ぎすぎて v0.1 の最小中核が肥大化することである。また、文脈を守るための基盤が、地域、企業、個人の文脈を集める中央管理DBへ変質する危険もある。さらに、AI解釈を事実または恒久属性として保存してしまう設計は、文脈主権に反する。

次回更新方針は、v0.1 / v0.2 / v0.3 の境界をさらに厳密化し、v0.1 については実装可能なデータスキーマ、CLI、保存ディレクトリ構成、JSONLイベント形式、Markdown frontmatter 仕様へ落とし込むことである。

## 20. 結論

Chronicle Stack が分散連合型のAI時代SNSになるための基本要求は、次の七点に集約される。

- ローカルに正本を置くこと。
- 共有は最小限にすること。
- 来歴を必ず残すこと。
- AIの変形を監査すること。
- 撤回と忘却を可能にすること。
- ノード同士を信頼と合意でつなぐこと。
- 投稿ではなく、問いと判断を流通させること。

この条件を満たすとき、Chronicle Stack は単なるSNSでも、単なるRAGでも、単なる文書管理でもなくなる。

それは、文脈主権を持った分散連合型の社会的記憶基盤である。

Chronicle Stack が目指すSNSとは、注意を奪うSNSではない。社会が自分の記憶を失わないためのSNSである。
