## Issue breakdown

### #TBD v1.7: Define local reviewer identity and session boundary for gated GUI mutation

Scope:

- define what reviewer identity fields are required when the local mutation gate is enabled
- define how session labels are represented and validated
- clarify what is descriptive metadata versus what is required for mutation acceptance

Repository-side progress so far:

- `ui_boundary`, `mutation_readiness`, blocked/apply responses, and list/detail read models now expose structured reviewer field requirements, reviewer-kind expectations, session-boundary status, `ui_intent` requirements, and shared expectation/authority notes
- current remaining work is design/ADR tightening for future mutation expansion rather than missing read-model visibility in this repo slice

Out of scope:

- multi-user authentication
- hosted identity providers
- authorization enforcement beyond local single-operator assumptions

### #TBD v1.7: Refine mutation-readiness blockers and enablement criteria

Scope:

- make `mutation_readiness` blockers operationally precise
- document explicit prerequisites for local write enablement
- keep overview, list, and detail readiness wording aligned

Repository-side progress so far:

- `mutation_readiness` now exposes blocker details, blocker-source summaries, operational readiness checks, scope notes, and row-level short summaries across overview/list/detail surfaces
- current remaining work is limited to future copy maintenance or additional write-path states, not missing prerequisite visibility in the current slice

Out of scope:

- enabling writes by default
- replacing CLI as the primary write path

### #TBD v1.7: Align blocked-route preview and CLI fallback contracts across all local UI surfaces

Scope:

- keep blocked preview, apply response, recovery command, and follow-up command contracts synchronized
- ensure review queue, summary jobs, and detail views share the same fail-closed vocabulary
- extend smoke coverage only where contract surfaces expand

Repository-side progress so far:

- blocked preview, apply response, recovery/follow-up command contracts, and thin list-level CLI hints are now aligned across review queue, summary jobs, runtime records, and detail drilldown surfaces
- current remaining work is maintenance when new surfaces are added, rather than an open contract gap in the current repo slice

Out of scope:

- real auth/authz enforcement
- new mutation capability beyond the existing explicit gate

### #TBD v1.7: Design explicit GUI review write routes and failure contracts

Scope:

- define route shapes for approve / reject / request-changes
- define request/response contracts for successful and failed local writes
- define how explicit local enablement is checked server-side

Repository-side progress so far:

- the local UI now exposes the write-route template, expected request fields, action set, and status-code contract summary on `ui_boundary` / `mutation_readiness`
- the repo now also exposes the same write-route and result-contract vocabulary on blocked preview, apply responses, list-level preview summaries, and detail drilldown surfaces
- the current repo still stops short of claiming authenticated or default-on GUI mutation semantics

Out of scope:

- default-on mutation
- unaudited write paths

### #TBD v1.7: Define browser-triggered rollback and error classification model

Scope:

- classify audit-write failure versus Chronicle append failure
- define rollback reporting rules for browser-triggered review actions
- align error classes with CLI/operator recovery guidance

Repository-side progress so far:

- fail-closed browser-triggered review responses now distinguish `mutation_disabled`, `audit_insertion_failed`, and `decision_persistence_failed`
- preview/apply/result surfaces now expose rollback status, possible error codes, recovery commands, follow-up commands, and CLI-equivalent hints in a shared contract shape
- current remaining work is semantic/ADR tightening for future mutation expansion, not missing recovery visibility in the current repo slice

Out of scope:

- proving distributed correctness
- cross-process transactional guarantees beyond the current local boundary
