# v1.7 Phase H UI Mutation Threat Model

Related: `docs/adr/0016-http-bridge-auth-boundary.md`, `docs/adr/0018-local-ui-read-only-navigation-boundary.md`, `docs/v1.7-phase-h-auth-ui-design.md`

## Purpose

This note captures the minimum threat model Chronicle Stack must satisfy before any write-capable local UI review flow is enabled.

## Current posture

- `chronicle ui` is read-only
- write routes are disabled
- review actions remain CLI-led
- local interactive state is ephemeral UI state only

## Primary threats

- shared-machine operator confusion causing unintended approval actions
- self-declared reviewer identity being mistaken for enforced identity
- future write routes bypassing audit insertion requirements
- convenience drift turning preview actions into mutation without explicit auth/authz design
- derived UI state being mistaken for durable approval state

## Required controls before GUI mutation

- explicit authentication boundary for local UI sessions
- explicit authorization rule for who may approve / reject / request changes
- audit insertion guarantee for every GUI review mutation
- CLI-to-UI parity checks for available review actions
- clear rollback / failure semantics for partial review mutation failures

## Non-goals

- hosted multi-user deployment design
- remote identity provider integration
- unattended approval automation
- replacing append-only CLI review as source-of-truth mutation path without further ADR work
