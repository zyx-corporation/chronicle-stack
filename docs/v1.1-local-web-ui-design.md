## Purpose

This document designs `chronicle ui`, an explicitly launched local web UI for Chronicle Stack.

The design goal is to improve human-facing review without weakening Chronicle Stack's local-first, inspectable, no-daemon-by-default architecture.

This document captures the v1.1 local-web-UI starting point.
For the broader target-state product UI and the implementation-gap mapping from that target back to current Chronicle surfaces, see the final UI design documents listed above.

## Design summary

`chronicle ui` is a foreground command that serves a local read-first UI over the current Chronicle root.

```bash
chronicle ui
```

Default behavior:

- host: `127.0.0.1`
- port: `8765`
- mode: read-only
- lifecycle: foreground process only
- browser opening: explicit option, not mandatory
- persistence: none beyond existing Chronicle files
- external calls: none

The process ends when the user terminates it in the terminal.

## Why explicit local web UI

The v1.1 static Review Console improves inspection but remains a generated artifact. A local web UI can provide richer navigation, filtering, and drill-down without turning Chronicle Stack into a hosted service.

The key architectural choice is explicit launch. `chronicle ui` must not install a daemon, autostart a service, or expose a network listener unless the user intentionally runs it.

## Command shape

Initial command proposal:

```bash
chronicle ui [OPTIONS]
```

Options:

```text
--host TEXT       Bind host. Default: 127.0.0.1.
--port INTEGER   Bind port. Default: 8765.
--open           Open browser after startup. Default: false.
--read-only      Force read-only mode. Default: true for v1.1.
--root PATH      Chronicle root. Default: current working directory.
--json           Print startup metadata as JSON.
```

The first implementation should reject write-capable mode if requested.

## Startup output

Human-readable startup output should include:

```text
Chronicle Stack local UI
Root: /path/to/project
Serving: http://127.0.0.1:8765
Mode: read-only
Boundary: no daemon, no external model API, no GraphRAG runtime, no vector DB, no graph DB
Press Ctrl-C to stop.
```

JSON startup output should include:

```json
{
  "host": "127.0.0.1",
  "port": 8765,
  "url": "http://127.0.0.1:8765",
  "root": "/path/to/project",
  "read_only": true,
  "runtime": "foreground-local-ui",
  "external_runtime": false
}
```

## Initial read surfaces

The first local UI should expose read-only views corresponding to the static Review Console:

- overview
- event timeline
- contexts
- artifacts
- decisions
- RDE diff records
- boundary rules
- audit events
- lifecycle markers
- package review snapshot
- graph-ready export summary
- local source path / provenance panels

## Internal endpoint sketch

The first implementation can keep endpoints minimal and local-only.

Candidate read endpoints:

```text
GET /              HTML shell
GET /api/overview  Chronicle summary and counts
GET /api/events    Recent Chronicle events
GET /api/contexts  Context records
GET /api/artifacts Artifact summaries
GET /api/decisions Decision records
GET /api/rde       RDE diff records
GET /api/boundary  Boundary rules
GET /api/audit     Audit events
GET /api/lifecycle Lifecycle markers
GET /api/package-review Package review snapshot
GET /api/graph-summary Graph node/edge counts
```

Endpoints should return data derived from local Chronicle files only.

## Framework boundary

Two implementation strategies are acceptable:

### Option A: Python standard library first

Use `http.server` or a small custom WSGI-style handler. This avoids dependencies and preserves installer simplicity.

Pros:

- no new dependency
- low supply-chain surface
- easy to explain

Cons:

- more manual routing / response handling
- less ergonomic if UI grows

### Option B: minimal local web dependency

Use a small dependency only if stdlib becomes too brittle.

Pros:

- cleaner routing and testing

Cons:

- new dependency and supply-chain surface
- installer impact

Recommended v1.1 initial choice: Option A unless implementation complexity becomes unreasonable.

## Security posture

`chronicle ui` is not an access-control layer. It is a local browser-facing review surface over local files.

Default controls:

- bind to `127.0.0.1`
- do not bind to `0.0.0.0` by default
- show the served root path in terminal and UI
- read-only by default
- no background persistence
- no external network calls
- no model API calls
- no GraphRAG / vector DB / graph DB runtime

If future versions allow non-loopback binding, they should require explicit `--host` and print a warning.

## Write boundary

v1.1 should be read-only.

Future write-capable UI actions require separate review and should start with preview-only flows, such as:

- preview export command
- preview package command
- preview classification update
- preview lifecycle marker

Actual writes should require explicit confirmation and should remain reproducible as CLI commands.

## Relationship to static Review Console

`chronicle export --format html` remains the static single-file Review Console.

`chronicle ui` is complementary:

- static Review Console: portable artifact, no server, easy to share internally
- local web UI: interactive local review, foreground only, not share-by-default

Neither surface is enforcement or proof.

## Relationship to Sayane and CSG-RAG

`chronicle ui` must not import Sayane or CSG-RAG runtime responsibilities.

It may inspect Chronicle-controlled packages and graph-ready exports, but it must not:

- execute Sayane reasoning loops
- call model APIs
- run GraphRAG retrieval
- manage vector stores
- manage graph databases

## Smoke test expectations

A minimal smoke test should verify:

- `chronicle ui --help` shows command help
- server can start on loopback using an ephemeral or test port
- `/api/overview` returns valid JSON
- `/` returns HTML
- process can shut down cleanly
- startup metadata reports read-only mode

## Release criteria impact

A future v1.1 release should include:

- local UI design doc
- local UI implementation
- smoke test coverage
- README usage section
- release notes preserving runtime boundaries

## Warning classification

- Runtime warning: `chronicle ui` must not become a hidden daemon or autostart service.
- Security warning: localhost UI is still browser-exposed and should be treated as a local risk surface.
- Semantics warning: UI review states are not enforcement or correctness proof.
- Architecture warning: local UI must not absorb Sayane, CSG-RAG, GraphRAG, vector DB, or graph DB responsibilities.

## RDE review

### Preserved

- Chronicle Stack remains local-first.
- CLI remains canonical and scriptable.
- Static Review Console remains available.
- Advisory/diagnostic semantics remain explicit.

### Transformed

- Human-facing inspection moves from static export to explicit local interactive review.

### Supplemented

- `chronicle ui` command boundary.
- Loopback-only default.
- Read-only first release rule.
- Endpoint and smoke-test sketch.

### Unresolved

- Exact implementation module layout.
- Whether stdlib is sufficient for all routing needs.
- Whether later write-capable UI actions belong in Chronicle Stack core.

### Deviation risks

- Hidden background service creep.
- Public network exposure by convenience.
- Treating UI visibility as access control.
- Turning Chronicle Stack into a hosted app prematurely.
