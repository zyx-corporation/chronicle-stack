## Issue breakdown

### #TBD v1.9: Refine local reviewer/session enforcement wording across UI and release docs

Scope:

- clarify what current local reviewer/session fields enforce versus what they only describe
- keep wording aligned across ADRs, release docs, and UI-facing contract descriptions
- preserve the explicit local single-operator boundary

Current repository-side progress in this issue:

- `reviewer_enforcement_summary` now exposes route-enforced request fields, eligible reviewer kinds, and read-only scope notes
- README and release status now state explicitly that read-only UI surfaces do not grant authority on their own
- `reviewer_validation_gate_summary` now exposes shared validation/gate failure families so readiness, preview, apply, and recovery surfaces stay aligned

Out of scope:

- hosted identity providers
- multi-user-safe authority enforcement

### #TBD v1.9: Align reviewer/session validation and gate language across route/readiness surfaces

Scope:

- keep reviewer/session gate wording aligned with existing route and readiness contracts
- reduce wording drift between preview, apply, readiness, and release-facing surfaces
- preserve fail-closed semantics and current CLI parity expectations

Out of scope:

- default-on mutation
- non-local review operators

### #TBD v1.9: Retire remaining v1.8-centric active-release wording once v1.9 surfaces exist

Scope:

- move the active-release vocabulary from `v1.8.0` to `v1.9.0` as new docs and implementation land

Out of scope:

- deleting `v1.8.0` release records
